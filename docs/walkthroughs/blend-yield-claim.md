# Walkthrough 1: Blend Yield Claim + USDC Conversion

**Use case:** Delegate to an AI agent the ability to claim yield from a Blend pool and convert it to USDC, without giving the agent full account control.

**Involved protocols:** Blend Protocol (lending/borrowing + yield distribution), Soroswap or Phoenix DEX (swap)

---

## 1. Scenario

You have supplied USDC to a Blend pool. The pool accumulates BLEND governance token rewards over time. You want an agent to:

1. Call `claim()` on the Blend pool to receive BLEND tokens
2. Swap BLEND for USDC via a DEX router
3. Do nothing else — no withdrawals, no transfers to other addresses

**Security requirement:** The agent can only call these two specific functions. It cannot withdraw principal, cannot send funds to any address other than your own account, cannot trade other assets.

## 3. Policy Synthesis

```bash
curl -X POST http://localhost:3000/mcp \
  -d '{ "tool": "synthesize_policy", "input": { "session_id": "sess_abc123", "smart_account_id": "CACCOUNT_USER..." } }'
```

The synthesizer produces:

### Context Rule

```json
{
  "label": "Blend Yield Claim + USDC Swap Delegation",
  "scope": [
    { "contractId": "CBENDPOOL...", "functionName": "claim" },
    { "contractId": "CSOROSWAP_ROUTER...", "functionName": "swap_exact_tokens_for_tokens" }
  ],
  "lifetime": {
    "type": "duration_seconds",
    "durationSeconds": 7776000,
    "description": "90 days from installation"
  }
}
```

### Proposed Policies

1. **OZ Spending Limit — BLEND** (standard OZ policy)
   - Asset: BLEND token
   - Limit: 50,000,000 (7 decimals = 5 BLEND) per day
   - Rationale: Observed max outbound BLEND = 50,000,000. Caps the agent's ability to drain BLEND.

2. **Custom Call Filter** (generated Rust)
   - Allowed: `swap_exact_tokens_for_tokens` only with path `[BLEND, USDC]` as first 3 args
   - Rationale: Prevents swapping to any token other than USDC.

### Clarifying Questions

> Q1: "I observed 5 BLEND transferred (50,000,000 in base units). Should the daily spending cap be 5 BLEND (tight) or a higher budget?"
> → User answers: "Allow up to 20 BLEND per day (agents run at unpredictable times)"

> Q2: "Should the swap path be locked to BLEND→USDC only, or allow any output token?"
> → User answers: "Lock to USDC only"

> Q3: "How long should this delegation be active?"
> → User answers: "90 days"

## 7. Security Analysis

| Attack vector | Prevented by |
|---------------|--------------|
| Drain principal from Blend pool | Scope doesn't include `withdraw()` |
| Transfer BLEND to attacker | Spending limit + call filter pin recipient |
| Swap BLEND for non-USDC | Call filter checks path[1] == USDC |
| Claim more than budget | Spending limit (20 BLEND/day) |
| Use after delegation expires | Time-bound lifetime (90 days) |
| Drain other assets (XLM, etc.) | Only BLEND in spending limit scope |
